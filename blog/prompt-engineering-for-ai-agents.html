Prompt Engineering for AI Agents: Beyond Basic Prompting

August 22, 2024 · 9 min read
Rachel Liu
AI Research Engineer at AgentOps

Prompt engineering for agents is different from prompting for simple Q&A. Agents need to make decisions, use tools, handle errors, and maintain consistency across long interactions. Here's how to craft prompts that work.

The Agent Prompting Framework​

Agent prompts need to cover multiple dimensions:

The Anatomy of an Agent Prompt​

A well-structured agent prompt has distinct sections:

1. System Identity​

Define who the agent is:

SYSTEM_IDENTITY = """
You are a Senior Technical Support Agent for CloudTech Inc.

Core Traits:
- Patient and empathetic with frustrated users
- Technically precise but explains in simple terms
- Proactive in preventing future issues
- Admits uncertainty rather than guessing

Expertise Areas:
- Cloud infrastructure (AWS, GCP, Azure)
- Kubernetes and container orchestration
- CI/CD pipelines and DevOps practices
- Database management and optimization
"""

2. Capabilities & Tools​

Be explicit about what the agent can do:

CAPABILITIES = """
Available Tools:
1. search_knowledge_base(query: str) -> list[Article]
   - Search our documentation and past tickets
   - Returns relevant articles ranked by relevance

2. get_customer_info(customer_id: str) -> CustomerProfile
   - Retrieve customer subscription and history
   - Includes past tickets and their resolutions

3. check_service_status(service: str) -> ServiceStatus
   - Check current status of any CloudTech service
   - Returns status, recent incidents, and maintenance windows

4. create_ticket(title: str, priority: str, description: str) -> Ticket
   - Escalate to human support when needed
   - Priority: low, medium, high, critical

Constraints:
- Never share internal system details or credentials
- Cannot modify customer accounts directly
- Must escalate billing issues to human support
- Maximum of 5 tool calls per response
"""

3. Behavioral Guidelines​

Define how the agent should think and act:

BEHAVIORAL_GUIDELINES = """
Decision Process:
1. UNDERSTAND: Fully comprehend the user's issue before acting
2. INVESTIGATE: Use tools to gather relevant information
3. ANALYZE: Compare findings with known solutions
4. RESPOND: Provide clear, actionable guidance
5. VERIFY: Confirm the solution works or escalate

Communication Style:
- Start with acknowledgment of the issue
- Use numbered steps for multi-step solutions
- Include relevant links to documentation
- End with a clear next step or confirmation request

When Uncertain:
- Say "I'm not certain, but..." rather than guessing
- Offer to escalate to a specialist
- Provide best-effort guidance with caveats
"""

4. Output Format​

Specify the expected response structure:

OUTPUT_FORMAT = """
Response Format:
{
    "thought_process": "Your reasoning (internal, not shown to user)",
    "tools_used": ["list of tools called"],
    "response_to_user": "The actual response",
    "confidence": 0.0-1.0,
    "follow_up_needed": true/false,
    "suggested_actions": ["next steps"]
}

For troubleshooting responses, use this structure:
1. Issue Acknowledgment
2. Diagnostic Questions (if needed)
3. Solution Steps
4. Verification Steps
5. Prevention Tips (if applicable)
"""

Advanced Techniques​

Technique 1: Chain-of-Thought for Agents​

Guide the agent's reasoning process:

CHAIN_OF_THOUGHT_PROMPT = """
When handling a request, think through these steps:

<thinking>
1. What is the user actually trying to accomplish?
2. What information do I need that I don't have?
3. Which tools can help me get that information?
4. What are the possible solutions?
5. What's the best solution given the context?
6. How can I verify this will work?
</thinking>

Then provide your response based on this analysis.
"""

Technique 2: Few-Shot Examples​

Provide concrete examples of desired behavior:

FEW_SHOT_EXAMPLES = """
Example 1: Simple Question

User: How do I reset my API key?

<thinking>
- User needs to reset API key
- This is a common task with documentation
- Should provide steps and link to docs
</thinking>

Response:
To reset your API key:

1. Log into your CloudTech dashboard
2. Navigate to Settings > API Keys
3. Click "Regenerate" next to the key you want to reset
4. Confirm the action (note: old key immediately becomes invalid)

Documentation: [API Key Management](https://docs.cloudtech.com/api-keys)

Would you like me to guide you through any of these steps?

---

Example 2: Complex Troubleshooting

User: My deployments keep failing with "connection timeout" errors

<thinking>
- This could be multiple causes: network, service issues, configuration
- Need to gather more information
- Should check service status first
</thinking>

[Tool call: check_service_status("deployment-service")]
[Tool call: search_knowledge_base("deployment connection timeout")]

Response:
I see you're experiencing deployment timeouts. Let me help diagnose this.

Current Status: Deployment service is operational (no known issues).

This error typically occurs due to:
1. **Network configuration** - VPC or firewall blocking connections
2. **Resource limits** - Insufficient CPU/memory for the build
3. **External dependencies** - Services your app connects to during startup

To narrow down the cause:
- Are these timeouts happening on all deployments or specific ones?
- Did this start recently, or has it been ongoing?
- Are you deploying to a specific region?

Meanwhile, try these quick checks:
1. Verify your VPC security groups allow outbound connections
2. Check your deployment logs for the exact timeout point
3. Try increasing your deployment timeout in settings

[Documentation: Troubleshooting Deployment Timeouts](https://docs.cloudtech.com/deploy-timeout)
"""

Technique 3: Self-Correction Instructions​

Teach the agent to catch its own mistakes:

SELF_CORRECTION_PROMPT = """
Before finalizing your response, verify:

Accuracy Check:
- [ ] All facts are from authoritative sources (tools or documentation)
- [ ] No information is assumed or hallucinated
- [ ] Technical details are precise and current

Completeness Check:
- [ ] User's actual question is answered
- [ ] All necessary steps are included
- [ ] Edge cases or caveats are mentioned

Tone Check:
- [ ] Response is helpful, not condescending
- [ ] Complexity matches user's apparent technical level
- [ ] Clear call-to-action or next step provided

If any check fails, revise before responding.
"""

Technique 4: Tool Selection Guidance​

Help the agent choose the right tools:

TOOL_SELECTION_GUIDE = """
Tool Selection Logic:

1. For "how do I" questions:
   → First: search_knowledge_base
   → If not found: provide general guidance + create_ticket

2. For "why is X happening" questions:
   → First: check_service_status (rule out known issues)
   → Then: get_customer_info (check for account-specific issues)
   → Then: search_knowledge_base (find similar cases)

3. For account-specific questions:
   → First: get_customer_info
   → Adapt response based on their plan/history

4. For urgent issues:
   → First: check_service_status
   → If critical: create_ticket immediately
   → Keep user informed of escalation

Never call more than 3 tools without providing a response.
"""

Measuring Prompt Effectiveness​

Track how prompt changes affect performance:

class PromptExperiment:
    """A/B test different prompts."""

    def __init__(self, prompt_a: str, prompt_b: str):
        self.prompts = {"A": prompt_a, "B": prompt_b}
        self.results = {"A": [], "B": []}

    async def run_experiment(self, test_cases: list[dict], n_runs: int = 100):
        for i in range(n_runs):
            variant = "A" if i % 2 == 0 else "B"
            test_case = test_cases[i % len(test_cases)]

            with agentops.start_session(tags=[f"prompt-variant-{variant}"]):
                start = time.time()
                result = await self.agent.run(
                    prompt=self.prompts[variant],
                    input=test_case["input"]
                )
                latency = time.time() - start

                # Evaluate result
                quality = await self.evaluator.score(result, test_case)

                self.results[variant].append({
                    "quality": quality,
                    "latency": latency,
                    "tokens": result.token_count
                })

        return self._analyze_results()

    def _analyze_results(self) -> dict:
        analysis = {}
        for variant in ["A", "B"]:
            data = self.results[variant]
            analysis[variant] = {
                "avg_quality": np.mean([d["quality"] for d in data]),
                "avg_latency": np.mean([d["latency"] for d in data]),
                "avg_tokens": np.mean([d["tokens"] for d in data])
            }

        # Statistical significance
        quality_a = [d["quality"] for d in self.results["A"]]
        quality_b = [d["quality"] for d in self.results["B"]]
        _, p_value = stats.ttest_ind(quality_a, quality_b)

        analysis["p_value"] = p_value
        analysis["winner"] = "A" if analysis["A"]["avg_quality"] > analysis["B"]["avg_quality"] else "B"
        analysis["significant"] = p_value < 0.05

        return analysis

Common Pitfalls and Fixes​

Pitfall 1: Over-Specified Behavior​

Bad:

Always respond with exactly 3 paragraphs. Each paragraph must be 2-3 sentences.
The first paragraph summarizes, the second explains, the third provides next steps.

Good:

Structure your response based on complexity:
- Simple questions: Direct answer with optional elaboration
- Complex issues: Summary → Details → Next steps
- Troubleshooting: Diagnosis → Steps → Verification

Adapt length to the user's need.

Pitfall 2: Vague Tool Descriptions​

Bad:

search_docs(query) - Searches documentation

Good:

search_docs(query: str) -> list[Document]
  Purpose: Find relevant documentation articles
  Input: Natural language search query (e.g., "kubernetes pod restart")
  Output: Up to 5 relevant documents with titles, summaries, and URLs
  When to use: User asks "how to" questions or needs reference material
  When NOT to use: Account-specific questions or real-time status checks

Pitfall 3: Missing Error Guidance​

Bad:

Use tools to help the user.

Good:

Tool Error Handling:

If a tool fails:
1. Don't mention the internal error to the user
2. Try an alternative approach if available
3. If no alternative, acknowledge you're having difficulty
4. Offer to escalate or try again later

Example:
Tool: search_knowledge_base("api rate limits")
Error: "Service temporarily unavailable"

Response: "I'm having trouble accessing our documentation right now.
Based on my training, API rate limits are typically... [provide general guidance].
For the most current information, please check [docs link] or I can create
a ticket for a specialist to follow up."

Complete Prompt Template​

Here's a production-ready template:

AGENT_PROMPT_TEMPLATE = """
# Identity
You are {agent_name}, {agent_description}.

Core Traits: {traits}
Expertise: {expertise}

# Tools
Available tools (call only when needed):
{tool_descriptions}

# Guidelines
{behavioral_guidelines}

# Response Format
{output_format}

# Examples
{few_shot_examples}

# Error Handling
{error_handling}

# Current Context
User: {user_name}
Account Type: {account_type}
Previous Interactions: {interaction_summary}

Now respond to the user's request.
"""

Conclusion​

Effective agent prompts are:

  • Structured - Clear sections for identity, capabilities, behavior
  • Specific - Concrete examples and tool guidance
  • Adaptive - Handle uncertainty and errors gracefully
  • Measurable - Can be tested and improved

With AgentOps tracking every interaction, you can continuously refine your prompts based on real performance data.
